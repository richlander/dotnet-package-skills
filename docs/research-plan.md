# Research plan & status

Status snapshot and forward plan for the package-grounding study. Written as a cross-session
handoff: read this first when resuming. Last updated 2026-06-21.

## POV (the thing we are arguing)

A NuGet package can ship a short, agent-targeted `AGENTS.md` ("grounding") that makes it
self-teaching for AI agents. Grounding is *pushed* (ships with the package, invisible to the user,
not removable) — distinct from a *skill*, which is *pulled* (opt-in, user-visible). The value of
grounding concentrates in the **non-resident delta**: version breakages and silent, compile-clean
gotchas the model is *not* already trained on. General API usage is model-resident and gains little.
Because strong models are already resident on most of this, grounding is a **distillation** play:
author with a strong model, deliver value to weaker/cheaper models, and clear a **Pareto gate** — help
Haiku/Sonnet without harming (or paying more for) Opus. The recommended delivery vehicle is a
skill-inspired, progressively-projected NuGet MCP: a **resident index** (the `AGENTS.md` frontmatter
`description`, scoped to the restored project's *direct* dependencies) plus the **body fetched on
demand**.

## The core limitation: evidence is shallow, and the method risks overfitting

We have a few prompts per package, and the **metric (IET weights) and thresholds were tuned on the
same prompts we report on**. That is hypothesis-generating, not confirming. Fixing believability is a
**methodology** problem first, a **volume** problem second.

## Workstream B — prompt corpus (the believability gap)

**Due-diligence minimum (agreed scope): 12 packages × 12 prompts each = 144 prompts.** Authored from
real-world sources (GitHub issues, Stack Overflow, release notes) — *not* synthetically, and
respecting the circularity caveat: do not let one model author the gotcha, the test, and the answer.

**Package slate (12)** — three deliberate ingredients:

- **Out-of-training anchors (2)** — `markout` and `NuGetFetch` (both author-controlled, *not* in any
  model's training data). These are the cleanest possible test: any competence must come from the
  grounding, not residency. They bound the "pure net-new value" end of the spectrum.
- **Sibling cluster in `Microsoft.Extensions.*` (5)** — `Microsoft.Extensions.AI`,
  `.DependencyInjection`, `.Logging`, `.Hosting`, `.Configuration`. These co-occur constantly as
  direct dependencies, so they specifically test **how sibling `AGENTS.md` descriptions merge in the
  resident index** — the multi-dependency projection behavior, not just single-package value.
- **Worked examples + community diversity (5)** — `System.Text.Json`, `System.CommandLine` (existing
  worked gotchas), plus `Polly` (v8 redesign), `Microsoft.EntityFrameworkCore`, and `Serilog`
  (top-downloaded, real version-delta gotchas).

*(Slate is a strong default, adjustable — the constraints are: include markout + NuGetFetch, and a
`Microsoft.Extensions.*` sibling cluster for the merge demonstration.)*

**Split into a dev set (tune metric/thresholds/descriptions) and a held-out test set (the reported
numbers).** This is the single most important change for credibility.

Prompt taxonomy (✦ = added beyond the original list):

- Greenfield vs **brownfield** (modify existing code).
- Feature discovery, **open-ended** ("what features should I consider from the latest version?").
- Feature discovery, **closed** ("use this named feature").
- **Scenario-implied** — the feature is required by implication ("enable Native AOT" for STJ;
  "upgrade to the latest version").
- ✦ **Harm / no-op controls** — the package's gotcha is irrelevant; measures the *no-harm* half of
  the Pareto claim (token cost, derailment) and self-selection false positives.
- ✦ **Distractor / wrong-package** — the resident description must *not* fire (self-selection precision).
- ✦ **Brownfield bug-repair** — the footgun encountered, not announced ("this returns null, fix it"
  = STJ case-sensitivity in the wild). Higher fidelity than feature prompts.
- ✦ **Resident-knowledge** — things the model already knows cold; expect zero value, must not harm
  (the Pareto control).
- ✦ **Version-pinned-old vs latest** — gotchas are version-deltas, so version context is a treatment
  variable, not a constant.
- ✦ **Multi-dependency** — realistic tasks touching several packages (resident-index scaling).

Each prompt needs a **gradeable success criterion** (judge rubric + ground-truth key), version pin,
and fixtures. Today the harness (`eng/run-channel-matrix.sh`) is hardcoded per task via a bash
`case` — **generalize to data-driven task definitions** before scaling to dozens of prompts.

## Workstream C — analysis design (two distinct kinds)

1. **Exploratory** (hypothesis-generating): define/validate a valuable metric, characterize agent
   behavior, find whether NuGet-MCP pivots add value. Run on the **dev set**. Free to iterate.
2. **Confirmatory** (the believable result): does `AGENTS.md` provide **value and no harm** with the
   infra we have today? Run on the **held-out test set** with the metric **frozen**. Pre-commit the
   arms and success criteria.

Cross-cutting requirements:

- **Model spectrum (agreed): 5 models, two providers.** Three from one provider to form a
  capability spectrum — **Anthropic Haiku / Sonnet / Opus** — plus two from another provider —
  **OpenAI GPT-5.5 and a GPT mini** (e.g. GPT-5 mini). The within-provider spectrum demonstrates the
  distillation/Pareto story (value on weak models, no harm on Opus); the second provider guards
  against single-provider overfit and tests whether grounding generalizes across model families.
- **Cost budget & sampling** — the full grid is 12 packages × 12 prompts × 5 models × ~3–5 channels ×
  N runs. That is large and Opus/GPT-5.5-expensive; pick a sampling/stratification strategy up front
  (e.g. full grid on the cheap models, a stratified subsample on the expensive ones).
- **Reproducibility** — pinned package versions, fixed judge model, clean cache-injection hygiene
  (the repo `grounding/*/AGENTS.md` is separate from the `~/.nuget` cache copy the eval injects).

## Workstream D — deliverable

  MCP serving it? Both? The PR target depends on this answer.

  argues for but does not yet build.
